# 开发与更新规范

## 1. 基本原则

- 不得随意修改代码。
- 只允许更改与当前需求直接相关的部分。
- 不得借需求之名进行无关重构、无关样式调整、无关依赖更新或无关文件整理。
- 如果发现需求描述不清、实现方向与需求可能不一致，或需要改变现有功能、结构、接口、数据、样式、流程，必须先询问并获得确认。
- 未经确认，不得擅自扩大改动范围。

## 2. 代码改动范围

- 每次更新前必须明确本次需求、影响范围和预期结果。
- 修改前应先阅读相关模块，理解现有逻辑后再动手。
- 优先保持现有项目结构、命名风格、代码风格和技术方案。
- 不得删除、覆盖或回退他人已有改动，除非获得明确授权。
- 对公共模块、核心流程、数据库结构、接口协议、权限逻辑等高影响区域的改动，必须特别谨慎，并提前说明风险。

## 3. 架构与代码质量

- 代码应规范化、模块化，并适合长期稳定维护。
- 新增或调整功能应优先采用清晰的 OOP 组件化设计。
- 模块应职责单一，避免把多个不相关功能混在一起。
- 组件、类、方法、接口应命名清晰，便于后续维护和扩展。
- 避免重复代码；必要时抽象为稳定的公共方法、服务类或组件。
- 不得为了短期完成需求引入难以维护的临时代码。
- 对复杂逻辑应添加必要注释，但不得添加无意义注释。

## 4. 需求确认机制

以下情况必须先询问，不能擅自决定：

- 需求与现有功能、设计或数据结构冲突。
- 需要改变用户可见的页面、交互、文案、样式或流程。
- 需要改变接口字段、返回格式、数据库结构或第三方集成逻辑。
- 需要删除旧功能、旧文件、旧配置或旧依赖。
- 需要引入新依赖、新框架、新服务或新构建流程。
- 发现需求存在多种实现方案，且不同方案会影响后续维护。

## 5. 测试要求

- 每次更新必须通过自动化测试后，才允许进入下一步。
- 自动化测试包括但不限于：
  - 单元测试
  - 集成测试
  - 构建测试
  - CI/CD 流水线测试
- 如果项目已有 CI/CD 流程，更新必须保证 CI/CD 全部通过。
- 如果当前项目暂未配置自动化测试，应记录未配置原因，并说明已完成的替代验证方式。
- 自动化测试通过后，还必须完成必要的人工测试。
- 未通过人工测试，不得上传或合并到 GitHub。

## 6. 人工测试要求

- 人工测试应覆盖本次需求的主要使用流程。
- 人工测试应确认：
  - 功能符合需求。
  - 页面或接口行为符合预期。
  - 原有关键功能未被破坏。
  - 没有明显的报错、异常、布局问题或数据错误。
- 人工测试结果必须记录在更新日志中。

## 7. 更新日志要求

每次更新必须写日志。日志应至少包含：

- 更新时间。
- 更新人或执行人。
- 需求来源或任务说明。
- 修改的主要内容。
- 涉及的文件或模块。
- 自动化测试结果。
- 人工测试结果。
- 是否已上传 GitHub。
- 已知问题或后续待办。

建议日志格式：

```markdown
## YYYY-MM-DD 更新记录

- 需求说明：
- 修改内容：
- 涉及模块：
- 自动化测试：
- 人工测试：
- GitHub 状态：
- 已知问题：
```

## 8. GitHub 更新规范

- 每完成一个独立项目、功能或修复，应优先更新到 GitHub。
- 上传 GitHub 前必须确认：
  - 代码改动范围符合需求。
  - 自动化测试已通过。
  - 人工测试已通过。
  - 更新日志已填写。
  - 没有提交无关文件、临时文件、敏感信息或本地配置。
- Git 提交信息应清晰描述本次改动内容。
- 不得将未测试、未确认或有明显风险的代码直接上传 GitHub。

## 9. 交付标准

一次更新完成后，应满足以下条件：

- 需求已实现，且未擅自改变需求外内容。
- 代码结构清晰，符合模块化和 OOP 组件化要求。
- 自动化测试通过。
- 人工测试通过。
- 更新日志完整。
- GitHub 已按规范更新，或明确记录未更新原因。

## 10. 违规处理

如出现以下情况，应停止继续更新并立即确认：

- 改动范围超出原始需求。
- 测试失败且原因未解决。
- 人工测试发现严重问题。
- 需要破坏兼容性或改变核心设计。
- 无法判断某项改动是否符合需求。

任何不确定的地方，必须先问清楚，再继续执行。
